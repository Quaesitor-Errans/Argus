# Database Migrations

## Purpose

Argus uses Alembic to version and evolve its SQLAlchemy database schema.

Application services must not create or alter database tables. Collection,
parsing, and analysis services operate only after the schema has been brought
to the current revision.

The CLI applies pending migrations before executing an application command.
Schema migration remains separate from session and repository lifecycle
management.

## New database

A new database can be initialized explicitly with:


python -m alembic upgrade head
Running an Argus CLI command also applies pending migrations automatically:

python main.py collect

Base.metadata.create_all() is not used for application schema management.
## Existing pre-Alembic database

The initial Alembic revision represents the schema used before migration
infrastructure was introduced.

An existing database that already has this exact schema must be backed up and
then marked with:


python -m alembic stamp head

stamp records the revision without executing schema operations.

It must not be used to conceal an unknown or incompatible schema. The existing
tables, constraints, indexes, and enum representations must first be verified
against the initial migration.

After stamping or upgrading, verify the database with:

python -m alembic current
python -m alembic check

## Creating migrations

When SQLAlchemy models change:

1. update the ORM models;
2. generate a candidate migration;
3. review every generated operation;
4. test upgrade and downgrade paths;
5. verify that Alembic detects no remaining schema differences.

Generate a candidate with:


python -m alembic revision --autogenerate -m "describe schema change"

Autogenerated migrations are not accepted without manual review. Data
migration, constraint behavior, SQLite batch operations, downgrade safety, and
operation ordering must be checked explicitly.

Isolated migration testing

Migration tests may override the database URL with:

ARGUS_ALEMBIC_DATABASE_URL

This override is intended for Alembic execution and automated tests. It does
not replace normal application database configuration.

The test suite creates a temporary SQLite database, upgrades it to head, and
verifies that the expected schema exists.

Safety rules
Back up persistent databases before manual migration work.
Never run a baseline upgrade over tables that already exist.
Never stamp an unverified schema.
Never edit an applied revision after it has been shared.
Add a new revision for every subsequent schema change.
Keep migration scripts in version control.
